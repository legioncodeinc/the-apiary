# RFC 9113 - HTTP/2 (June 2022)

## Summary
RFC 9113 obsoletes RFC 7540 and defines HTTP/2. Core features: multiplexing multiple requests over a single TCP connection via streams (solving the HTTP/1.1 head-of-line-blocking problem at the application layer, but not at TCP transport layer); HPACK header compression (RFC 7541) which reduces header overhead significantly; stream prioritization; server push (allowed but deprecated in practice); HTTP/2 runs over TLS (h2) or plaintext (h2c, not supported by browsers). HTTP/2 uses binary framing instead of text-based HTTP/1.1 messages. Each stream has a stream ID; client-initiated streams use odd IDs, server-initiated (push) use even IDs. The connection preface is "PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n" followed by a SETTINGS frame.

## Key quotations / statistics
- "HTTP/2 enables a more efficient use of network resources and reduced perception of latency by introducing header field compression and allowing multiple concurrent exchanges on the same connection." (§1)
- "A single HTTP/2 connection can contain multiple concurrently open streams, with either endpoint interleaving frames from multiple streams." (§5)
- HPACK compression: "Header fields used in HTTP/2 are the same as those defined in RFC 9110... HPACK allows for efficient compression of repeated header fields."
- TCP head-of-line blocking: HTTP/2 eliminates it at the HTTP layer but not at TCP. A single lost TCP packet blocks ALL streams on that connection — fixed by QUIC in HTTP/3.
- Server Push removal: Chrome removed support in April 2022; other browsers followed. RFC allows it; production consensus is to avoid it.
