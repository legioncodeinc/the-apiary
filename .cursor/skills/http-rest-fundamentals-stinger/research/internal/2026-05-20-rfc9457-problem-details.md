# RFC 9457 - Problem Details for HTTP APIs (July 2023)

## Summary
RFC 9457 obsoletes RFC 7807 and defines a standard machine-readable format for HTTP error responses: `application/problem+json` and `application/problem+xml`. The Problem Details object has five members: `type` (URI reference categorizing the problem, defaults to `about:blank`), `title` (human-readable summary), `status` (HTTP status code as number), `detail` (human-readable explanation specific to this occurrence), `instance` (URI identifying the specific occurrence). Extensions are allowed by adding additional members. This is now the standard way to format API error bodies.

## Key quotations / statistics
- "The `type` member is a URI reference [URI] that identifies the problem type." (§3.1.1) — when dereferenced, it ideally provides human-readable documentation.
- "The `status` member is a JSON number indicating the HTTP status code... generated by the origin server for this occurrence." (§3.1.5)
- "Consumers MUST use the `type` URI for identifying the problem type, not the `title`." (§3) — title can be localized, type is the stable identifier.
- "Implementations MUST NOT change the value of problem type URI upon a version change." — type URIs are stable identifiers.
- Content-Type must be `application/problem+json` for JSON error responses (not `application/json`).
