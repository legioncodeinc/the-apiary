## 1. `KnowledgeDocument` types (14)

```prisma
enum KnowledgeDocType {
  company_context
  customer_persona
  product_info
  methodology
  framework
  best_practice
  faq
  resource
  goal_mapping
  ideal_client_avatar
  offer_definition
  strategic_positioning
  referral_strategy
  business_profile  // member's own ‚Äî auto-generated by onboarding
}
```

### Priority in text-budget retrieval

| Priority | Types | Label prefix |
|---|---|---|
| 1 | `company_context` | `Company Context:` |
| 2 | `methodology`, `framework` | `Methodology:` |
| 3 | `goal_mapping`, `ideal_client_avatar`, `offer_definition`, `strategic_positioning`, `referral_strategy` | `Coach Knowledge:` |
| 4 | `business_profile` (member's own) | `Member Profile:` |

In vector search mode, the member's `business_profile` is always appended after the top-K passages.

## 2. The data model

```prisma
model KnowledgeDocument {
  id           String              @id @default(cuid())
  title        String
  documentType KnowledgeDocType    @map("document_type")
  body         String
  visibility   KnowledgeVisibility @default(organization)
  creatorId    String              @map("creator_id")
  tenantId     String              @map("tenant_id")
  isDeleted    Boolean             @default(false) @map("is_deleted")
  createdAt    DateTime            @default(now())
  updatedAt    DateTime            @updatedAt

  @@index([tenantId, documentType])
}

enum KnowledgeVisibility {
  private        // creator only
  team           // not yet used
  organization   // all members of the tenant (default)
  public         // future use
}
```

Soft delete (`isDeleted: true`) removes from queries and triggers Qdrant vector removal via `removeKnowledgeDocument()`.

## 3. The three retrieval strategies (priority order)

```mermaid
flowchart LR
  A[buildKnowledgeContext] --> B{Pinned doc IDs in knowledgeBaseId?}
  B -- Yes --> C[Fetch pinned docs from Postgres]
  C --> G[Inject into prompt]
  B -- No --> D{QDRANT_URL + COHERE_API_KEY set?}
  D -- No --> F[Text-budget fallback (8K char)]
  D -- Yes --> E{enableGraphRAG flag?}
  E -- Yes --> H[Hybrid GraphRAG + RRF]
  E -- No --> I[Vector search 20 ‚Üí rerank top-5]
  H --> G
  I --> G
  F --> G
```

### Strategy 1: Pinned docs (`AiCoachConfig.knowledgeBaseId` JSON array)

```typescript
const pinnedIds = JSON.parse(knowledgeBaseId);  // ["docId1", "docId2"]
```

- Fetched directly from Postgres in specified order.
- Concatenated into a `Coach Knowledge:` block + text-budget base context.
- Capped at 8,000 chars total.
- **Qdrant is NOT queried in this path.**

Empty array `[]` is valid ‚Äî signals "no pinned docs, use text-budget base only."
A non-`[`-prefix value is treated as a legacy `CourseModule.id` and uses the text-budget path.

### Strategy 2: Vector search (+ optional GraphRAG)

```
KNOWLEDGE_CANDIDATES = 20    // broad ANN candidates
KNOWLEDGE_TOP_N      = 5     // final passages after Cohere rerank-v3.5
EPISODIC_CANDIDATES  = 10    // past session summaries
EPISODIC_TOP_N       = 3     // top episodic after rerank
```

If `enableGraphRAG` flag is set in `PlatformConfig.systemPromptBlocks`, the GraphRAG path runs for ~50% of users (hash-based A/B). See `guides/11-graphrag.md`.

### Strategy 3: Text-budget fallback

Activates when:
- `QDRANT_URL` or `COHERE_API_KEY` not set
- Qdrant returns zero results (cold start)
- Vector search throws

```
TOTAL_CHAR_BUDGET = 8000

Sections fetched in priority order:
1. company_context docs (non-private)
2. methodology/framework docs linked to moduleId's knowledgeDocIds
3. coach-specific docs linked to moduleId's knowledgeDocIds
4. member's business_profile doc

Truncate last section at boundary with "...(truncated)".
```

## 5. Vector indexing ‚Äî `indexKnowledgeDocument()`

```typescript
indexKnowledgeDocument({ id, tenantId, title, body, documentType }).catch(console.error);
```

Fire-and-forget from admin CRUD routes. Silently skipped if `QDRANT_URL` or `COHERE_API_KEY` are not set (dev without Qdrant works without errors).

### Chunking

`chunkText()`: 500-character target chunks with 20% overlap (`start += Math.floor(maxChars * 0.8)`).

Each chunk indexed with title prepended: `"${title}\n${chunk}"` ‚Äî anchors each chunk to its parent doc context.

500 chars (~125 tokens) is **below** Cohere's optimal 512-token window. Trade-off: smaller Qdrant point counts vs precision. A migration to 512-token chunks with re-indexing would improve precision (tracked as a should-refactor with eval gate).

### Point payload

See `guides/09-vector-payload-schema.md`. Knowledge doc points use `user_id: "__system__"`, `agent_type: "system"`, `content_type: "knowledge_doc"`.

### Idempotent upsert

`client.upsert()` with `wait: true`. Random UUIDs as point IDs (NOT stable doc-chunk IDs).

**Gap:** Re-indexing a document does NOT remove old chunks ‚Äî it adds new ones. `removeKnowledgeDocument()` (filter-deletes by `source_document_id`) MUST be called before re-indexing. The admin `DELETE` route does this; the `PUT` route currently does NOT (issue tracked, must-fix).

## 6. The `business_profile` always-append rule

After top-K knowledge passages, the member's `business_profile` document is always appended:

```typescript
const profileDoc = await prisma.knowledgeDocument.findFirst({
  where: { creatorId: memberId, tenantId, documentType: "business_profile", isDeleted: false },
});
if (profileDoc) passages.push(`--- Member Profile: ${profileDoc.title} ---\n${profileDoc.body}`);
```

This is the member's own profile doc, auto-generated by the onboarding agent. Always present in coaching prompts.

## 7. Episodic memory in vector context

`buildVectorContext()` also retrieves past session summaries from `conversations-{tenantId}`:

```
filter: tenant_id, user_id, content_type IN ["session_summary", "episodic_summary"]
EPISODIC_CANDIDATES = 10
EPISODIC_TOP_N      = 3   // after Cohere rerank
```

Episodic results appended as `[COACHING HISTORY]` block after knowledge passages.

## 9. Module-linked knowledge

`CourseModule.knowledgeDocIds` is a string array of `KnowledgeDocument` IDs. When `buildKnowledgeContext()` is called with a module ID (legacy string format in `knowledgeBaseId`), the text-budget fallback restricts knowledge to those linked documents.

**Vector search path does NOT filter by module** ‚Äî it searches all knowledge documents for the tenant. By design (vector search provides better global retrieval).

## 10. Auxiliary-collection retrieval gap (always flag)

`indexLessonContent()` indexes academy lessons into `academy-{tenantId}`. The collection is populated when lessons are created/updated via `admin-academy.ts`.

**Gap:** `buildKnowledgeContext()` only searches `knowledge-{tenantId}`, NOT `academy-{tenantId}`. Academy content is indexed but not retrieved during coaching.

Tracked as [the host repo's tracker](host repo issue tracker). mind-worker-bee flags this on every RAG audit until closed. The fix: extend `buildVectorContext()` to query both collections and fuse results (e.g., separate top-3 from each, then rerank both together).

## 11. Admin endpoints

| Method | Path | Action |
|---|---|---|
| `POST` | `/api/admin/knowledge` | Create; triggers async `indexKnowledgeDocument()` |
| `PUT` | `/api/admin/knowledge/:id` | Update; triggers async `indexKnowledgeDocument()` (BUT does NOT call `removeKnowledgeDocument()` first ‚Äî leak) |
| `DELETE` | `/api/admin/knowledge/:id` | Soft-delete; triggers async `removeKnowledgeDocument()` |

All endpoints require `admin` role and are scoped to `request.user.tenantId`.
