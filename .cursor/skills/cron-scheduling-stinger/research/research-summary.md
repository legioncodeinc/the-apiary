## 5 most influential sources

### 1. `external/2026-05-20-vercel-cron-jobs-official-docs.md`
**Why it matters most:** Vercel is likely the primary deployment target for the stinger's user base (Next.js / serverless). This source contains the January 2026 plan limit update (100 cron jobs per project on all plans), the critical "no retry on failure" policy, the CRON_SECRET security pattern, explicit guidance on idempotency and Redis locking, and the "Hobby = daily only" constraint that drives the workaround section. This feeds `guides/01-platform-limits.md`, the `CRON_SECRET` security callout, and `examples/vercel-cron-happy-path.md`.

### 2. `external/2026-05-20-distributed-cron-exactly-once-patterns.md`
**Why it matters most:** This is the architectural core of the stinger. Cron at scale is a distributed systems problem, and this source covers all three patterns: Postgres advisory locks (database-friendly, no additional infrastructure), Redis SETNX (multi-region friendly, lock TTL must exceed max job runtime), and idempotency keys (correct model for at-least-once platforms like Vercel and queues). The fencing token pattern and the PgBouncer warning are production-grade details that distinguish this stinger from basic guides. Feeds `guides/02-distributed-cron-correctness.md` and `examples/distributed-duplicate-prevention.md`.

### 3. `external/2026-05-20-cron-timezone-dst-production-patterns.md`
**Why it matters most:** DST bugs are silent, seasonal, and occur exactly twice a year - they are the quintessential "works in testing, fails in production at 2 AM once a year" category. This source documents both failure modes (spring-forward skip, fall-back double), platform-specific behaviors (GitHub Actions advances on spring-forward per RFC 5545), and the canonical defense (UTC-first, idempotency keys with unique constraint on scheduled_at). Feeds `guides/03-timezone-dst-safety.md` entirely.

### 4. `external/2026-05-20-healthchecks-io-official-docs.md`
**Why it matters most:** The cron-scheduling-worker-bee's critical directive states "heartbeat monitoring is mandatory for business-critical jobs." Healthchecks.io is the canonical tool for this, and this source covers the full integration model: dead man's switch semantics, `/start`+success+`/fail` signals for duration measurement, grace time configuration, and self-hosting for air-gapped or compliance-bound environments. Feeds `guides/05-observability-monitoring.md` and all three example files.

### 5. `external/2026-05-20-github-actions-schedule-event-docs.md` + `external/2026-05-20-github-actions-schedule-drift-community.md`
**Why they matter most (combined):** GitHub Actions `schedule:` is used by most engineering teams and is simultaneously the most unreliable platform cron option. The combination of official docs (5-minute minimum, UTC + IANA timezone now supported, 60-day auto-disable in public repos) and community reports (drift of 22+ minutes, complete non-execution in January 2026) gives the stinger a complete picture to advise teams: use `schedule:` only with a `workflow_dispatch` fallback and a Healthchecks.io heartbeat to detect missed runs. Feeds `guides/01-platform-limits.md` and `examples/github-actions-drift-mitigation.md`.

## Open questions that survived research

1. **What is Cloudflare's effective CPU time limit per Cron Trigger on the free tier vs paid?** The docs say 10ms CPU on free / 50ms on paid, but community reports suggest Workers Unbound (paid) allows much higher CPU for scheduled work when combined with Workflows (Durable Objects). The exact boundary for "use plain Worker vs switch to Workflows" needs clarification from Cloudflare's limits page.

2. **Does GitHub Actions `schedule:` + `timezone:` IANA support cover fall-back handling?** The docs only document spring-forward behavior ("advance to next valid time"). Whether fall-back is handled (skip the second occurrence vs run twice) is undocumented. `stinger-forge` should recommend testing explicitly.

3. **pg_cron vs BullMQ for database-native scheduling: depth of coverage?** The Command Brief flags this as an open question. The research confirms both are production-grade tools (Citus/pg_cron for PostgreSQL-native, BullMQ repeatable jobs for Node.js/Redis). Recommend a brief "alternatives" section in `guides/01-platform-limits.md` rather than full coverage, matching the Command Brief's suggestion.

4. **Vercel max execution duration for cron jobs.** The docs say "identical to Vercel Functions limits" and reference `maxDuration` documentation, but do not state the actual number. Free plan: 10s; Pro plan: 60s; Enterprise: 300s. Stinger-forge should confirm by fetching the Vercel Functions limits page directly.

5. **Healthchecks.io vs Cronitor for free-tier coverage at scale.** Free tier comparison: Healthchecks.io (20 checks), Cronitor (5 monitors). For teams with more than 20 jobs, the cost model changes significantly. `stinger-forge` may want to add a cost guidance note.
