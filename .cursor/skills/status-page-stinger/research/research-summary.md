## 5 Most Influential Sources

### 1. `external/2026-05-20-platform-comparison-2026.md`
**URL:** https://www.augmentcode.com/tools/best-status-page-tools (AugmentCode, May 18, 2026 - most recent comparative source)

**Why it matters for stinger-forge:** This source provides the most current (May 2026) cross-platform comparison table with verified pricing, free tier limits, self-hosting status, and entry-paid prices for 8 tools. It is the primary data source for `guides/00-platform-selection.md`'s decision matrix. The comparison confirms the 2026 "four-option" decision tree: Statuspage (Atlassian ecosystem) / Better Stack (all-in-one) / Instatus (value + channel breadth) / Cachet (OSS self-hosted). A fifth option class (Uptime Kuma, Gatus) for teams needing monitoring + status in a single self-hosted container is now prominent.

### 2. `external/2026-05-20-better-stack-platform-pricing.md`
**URL:** https://betterstack.com/pricing + https://betterstack.com/community/comparisons/better-stack-vs-instatus/

**Why it matters for stinger-forge:** Better Stack's add-on pricing model is complex and stinger-forge needs to warn practitioners that a "fully enterprise" Better Stack status page can exceed $800/month in add-ons before the base subscription. The SMS inclusion model (unlimited per responder at $29/month) is a key differentiator vs Instatus (BYOC/self-provided) and Statuspage (per-message model). The Better Stack vs Instatus comparison article is the single richest source for understanding the platform positioning trade-offs and will directly inform the `guides/00-platform-selection.md` decision tree.

### 3. `external/2026-05-20-incident-communication-templates.md`
**URL:** https://runframe.io/blog/incident-stakeholder-communication-templates (Runframe, February 2026)

**Why it matters for stinger-forge:** This source consolidates the 2026 practitioner consensus on incident communication into concrete templates, cadence rules, and severity matrix. The five golden rules, the severity-mapped update cadence table (SEV0 = 15 min, SEV1 = 30-60 min, SEV2 = 60-120 min), and the copy-ready template set should be reproduced almost verbatim in `guides/02-incident-communication.md`. The "5-minute rule" and "next update time commitment" as non-negotiable elements are specifically called out as the largest drivers of user trust loss when violated.

### 4. `external/2026-05-20-cachet-v3-stability.md`
**URL:** https://github.com/cachethq/cachet + https://cachet.mintlify.app/v3.x/introduction

**Why it matters for stinger-forge:** Answers the Command Brief's open question directly: "Is Cachet v3 stable?" Answer as of May 2026: NO - v3.x is under active development, missing Subscribers/Notifications and Metrics features, not yet in formal GA release. v2.4.1 is the production-stable version (last release November 2023). The GNOME project uses v3.x in production, suggesting the core is functional, but it is NOT a safe recommendation for teams that need subscriber email notifications (the feature is absent from v3.x). This finding significantly affects the platform-selection guide's Cachet recommendation: "use v2.x for production; evaluate v3 when GA."

### 5. `external/2026-05-20-subscriber-notifications-channels.md`
**URL:** https://support.atlassian.com/statuspage/docs/enable-subscribers + https://instatus.com/help/status-page/notifications/sms

**Why it matters for stinger-forge:** Confirms the architectural split in SMS delivery across platforms: Statuspage handles SMS internally (included in paid plans, multi-provider redundancy via Mailgun/Twilio/Plivo), Better Stack bundles SMS per-responder, and Instatus uses BYOC (user must bring a Twilio/Vonage/etc. account). This has direct pricing and compliance implications - the Instatus BYOC model means the team is responsible for their own A2P 10DLC registration and carrier compliance. Also confirms the critical Statuspage limitation: SMS only fires on incident CREATE and RESOLVE, not intermediate updates.

## 5 Open Questions for stinger-forge to Resolve

1. **Cachet v3 license**: The GitHub API returned "NOASSERTION" for the v3.x license. The v2.x is BSD-3-Clause. stinger-forge should verify the v3.x license before recommending it for enterprise OSS use in `guides/00-platform-selection.md`.

2. **Instatus SMS and A2P 10DLC responsibility**: Instatus uses BYOC for SMS. Does the Instatus user bear full responsibility for Twilio A2P 10DLC registration, carrier compliance, and STOP keyword handling - or does Instatus manage some carrier compliance on behalf of users? This affects the subscriber-notifications guide's SMS setup instructions.

3. **Atlassian Statuspage Business plan pricing discrepancy**: Multiple sources cite $399/month while others cite $499/month for the Business tier. The official Atlassian pricing page (fetched directly) shows $399/month. stinger-forge should pin the official price and note it may change.

4. **Post-mortem visibility norms**: Research surfaced no clear industry consensus on whether post-mortems should be public (company blog), semi-public (linked from status page), or private (internal only). stinger-forge should make an opinionated recommendation in `guides/04-post-incident-discipline.md` with a brief rationale (suggested: default to public for B2B SaaS; private for regulated industries or incidents involving security vulnerabilities).

5. **Better Stack native PagerDuty integration for status pages**: Research confirmed Better Stack has monitoring-to-status-page automation built-in (no external PagerDuty needed), but did not confirm whether a direct PagerDuty-to-Better-Stack-status-page API integration exists for teams using PagerDuty as their primary alerting tool with Better Stack only for status pages. stinger-forge should verify this gap in `guides/05-automation-integration.md`.
