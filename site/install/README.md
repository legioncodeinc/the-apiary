# `get.theapiary.sh` — the Honeycomb install surface

A self-contained **Cloudflare Pages** project that serves the one-command Honeycomb
installer at **https://get.theapiary.sh** with content negotiation, plus an
"inspect before you pipe" page and published SHA-256 checksums.

This resolves the PRD-050a non-gating follow-up (vanity domain + published checksum +
inspect page). It **consumes** PRD-048 — the page installs the published npm package
`@legioncodeinc/honeycomb`; it does not publish it.

---

## What's here

```
site/install/
├── build.mjs              # DRY build: copies canonical scripts, hashes them, renders the page
├── index.template.html    # the "inspect before piping" page template (checksums injected at build)
├── functions/
│   └── index.js           # Pages Function: GET / content-negotiates shell-vs-browser
├── _headers               # pins text/plain + nosniff on /install.sh /install.ps1 /SHA256SUMS; CSP on the page
├── README.md              # this file
└── dist/                  # GENERATED by build.mjs — gitignored, never committed
    ├── install.sh         # copy of scripts/install/install.sh (single source of truth)
    ├── install.ps1        # copy of scripts/install/install.ps1
    ├── SHA256SUMS         # sha256sum -c — verifiable checksums of the two scripts above
    ├── _headers           # copied from site/install/_headers (Pages reads it from the OUTPUT dir)
    └── index.html         # rendered inspect page with live checksums
```

Deploy is automated by [`.github/workflows/deploy-install-site.yaml`](../../.github/workflows/deploy-install-site.yaml)
(GitHub Actions + Wrangler → the Cloudflare Pages project `honeycomb-install`). See "Deployment" below.

**Single source of truth:** the installer scripts live ONCE at
[`scripts/install/`](../../scripts/install/). `build.mjs` copies them into `dist/` and
computes the checksums from exactly those copies, so what's served is always byte-identical
to what's checksummed. Never edit the copies in `dist/`; edit `scripts/install/` and rebuild.

## Content negotiation (the `get.pnpm.io` / `sh.rustup.rs` pattern)

| Request | Served | Why |
|---|---|---|
| `GET /` from a **shell** (curl/wget/PowerShell — `Accept` lacks `text/html`, or a CLI User-Agent) | `install.sh` as `text/plain` | so `curl -fsSL https://get.theapiary.sh \| sh` works |
| `GET /` from a **browser** (`Accept: text/html`) | the inspect `index.html` | a human gets the readable page |
| `GET /install.sh`, `/install.ps1`, `/SHA256SUMS` | always `text/plain` (`_headers`) | read/verify before running; `irm …/install.ps1 \| iex` works |

The negotiation lives in `functions/index.js` (`onRequest`). It only governs `/`; the explicit
file paths are plain static assets governed by `_headers`. The Function re-wraps the root shell
response to **guarantee** `text/plain; charset=utf-8` regardless of asset defaults.

## Deployment (Model A — GitHub Actions + Wrangler direct upload)

The Cloudflare side is **already provisioned** (via API): a Direct-Upload Pages project
**`honeycomb-install`** (`honeycomb-install.pages.dev`, production branch `production`) with the
custom domain **`get.theapiary.sh`** attached. CI builds and deploys it — no Cloudflare dashboard
build settings, no Git-integration OAuth.

[`.github/workflows/deploy-install-site.yaml`](../../.github/workflows/deploy-install-site.yaml)
runs on a pushed **release tag** (`v*`) and on manual dispatch: it runs `node site/install/build.mjs`
then `wrangler pages deploy site/install/dist --project-name=honeycomb-install --branch=production`.
Deploying with `--branch=production` (matching the project's production branch) makes the tagged
content the **production** deployment that `get.theapiary.sh` serves — so a tag's bytes go live
immutably, even though Pages itself keys "production" off a branch name, not a tag.

### Security controls (defense-in-depth)

Because the installer is served at a public endpoint and users pipe it directly into `sh` or `iex`,
the deployment workflow implements multiple security layers to prevent unauthorized deployments:

1. **Protected environment**: The deploy job requires manual approval via the `production` GitHub
   environment (configured at Settings → Environments → production with required reviewers). Even
   if a `v*` tag is pushed, the workflow pauses and waits for a trusted maintainer to review and
   approve the deployment before any bytes reach Cloudflare.

2. **Branch ancestry verification**: The workflow verifies that the triggering tag is reachable from
   the protected `main` branch. This ensures the tag's commit has passed through main's required PR
   reviews and CI quality gates (`.github/rulesets/main-protection.json`). A tag pointing to an
   arbitrary commit not on main is rejected before the build runs.

3. **Immutable tag semantics**: Tags are immutable refs. Once a tag is pushed and deployed, re-pushing
   the same tag name is rejected by Git, preventing silent replacement of a known-good release.

These controls ensure that only vetted, reviewed installer scripts reach production. See `SECURITY.md`
for the full security policy.

### Three one-time operator steps

1. **Configure the protected `production` environment** (REQUIRED for security). At the repository,
   go to **Settings → Environments → New environment**, create an environment named `production`,
   and configure:
   - **Required reviewers**: Add at least one trusted maintainer (ideally 2+) who must approve
     every deployment before it proceeds to Cloudflare.
   - **Deployment branches**: Optionally restrict to `main` only (belt-and-suspenders; the workflow's
     branch ancestry check already enforces this, but the UI setting adds a second gate).
   
   Without required reviewers, the workflow still references the `production` environment, but it
   will not pause for human approval. This environment must stay protected; the installer must never
   deploy without human review. See `SECURITY.md`.

2. **Point `theapiary.sh` at Cloudflare** (the zone is currently `pending`). At the registrar,
   set the nameservers to the ones Cloudflare assigned this zone:
   ```
   felicity.ns.cloudflare.com
   marge.ns.cloudflare.com
   ```
   Until the zone is `active`, `get.theapiary.sh` won't resolve and the Pages custom domain stays
   `pending`. (Keeping the zone on Cloudflare DNS is also what you want for the apex website later —
   point `theapiary.sh` → wherever the site is hosted and `get` → Pages, both records in this zone.)

3. **Add the deploy token to GitHub.** Repo → Settings → Secrets and variables → Actions → New
   repository secret:
   ```
   Name:  CLOUDFLARE_API_TOKEN
   Value: <a Cloudflare API token scoped Account → Cloudflare Pages → Edit>
   ```
   Mint it at Cloudflare dashboard → My Profile → API Tokens → Create Token (the
   "Edit Cloudflare Workers" template works, or a custom token with just **Account · Cloudflare
   Pages · Edit**). The account ID (`893c7bdc0782f11d23ebd2da8926c132`) is **not** secret and is
   inlined in the workflow.

Then push a `v*` tag (or run the workflow manually) to cut the first deploy.

### Verify after deploy

```sh
# shell client gets the script as plain text
curl -fsSL https://get.theapiary.sh | head -3
curl -fsSLI https://get.theapiary.sh | grep -i content-type      # text/plain; charset=utf-8

# explicit paths are plain text + nosniff
curl -fsSLI https://get.theapiary.sh/install.ps1 | grep -i 'content-type\|nosniff'

# checksums verify against the served scripts
curl -fsSL -O https://get.theapiary.sh/install.sh
curl -fsSL -O https://get.theapiary.sh/SHA256SUMS
sha256sum -c SHA256SUMS                                            # install.sh: OK

# a browser Accept header gets the HTML page instead
curl -fsSL -H 'Accept: text/html' https://get.theapiary.sh | grep -i '<title>'
```

## Version pinning (release-tag triggered, not a moving branch)

Production deploys are cut **only from a pushed release tag** (the workflow's `on: push: tags: ['v*']`),
never automatically from `main`. Reasons:

1. **Immutable served script.** A deploy happens only when you tag, so the script served at
   `https://get.theapiary.sh` is frozen for that release — it can't silently change under users on
   an unrelated merge to `main`.
2. **Stable checksum.** `build.mjs` hashes the exact scripts at build time, so a given tag produces
   a stable `SHA256SUMS`. Anyone who recorded a checksum can re-verify it later. If production
   tracked `main`, every merge would rotate the published checksum and break "verify the hash I
   saved yesterday".

To cut a new install-surface release: push the tag (ideally the same tag you publish the npm
package from), e.g. `git tag v0.1.1 && git push origin v0.1.1`. The workflow builds from that tag's
checkout and `wrangler pages deploy … --branch=production` promotes it to the live deployment; the
checksum on the page updates to match the newly served scripts. (`workflow_dispatch` is available
for a manual re-deploy of the default branch when needed.)

## Product selection + install-time telemetry (the-apiary PRD-002)

The scripts this site serves ([`scripts/install/install.sh`](../../scripts/install/install.sh) /
[`install.ps1`](../../scripts/install/install.ps1)) accept `--products=`, `--profile=`,
`--license=`, `--code=`, and `--dry-run` — see install.sh's header comment for the full grammar,
precedence, and manifest-resolution behavior. This site adds one optional convenience on top:

- **`?combo=<name>` sugar** ([`functions/index.js`](./functions/index.js)'s `COMBO_PRESETS`
  table): `https://get.theapiary.sh/?combo=full` streams the SAME `install.sh` bytes with an
  `export HONEYCOMB_INSTALL_PRODUCTS=...` line prepended — the exact env var the script already
  reads, just pre-filled. This is sugar, never the primary mechanism (ADR-0002); the response is
  served `Cache-Control: no-store` and is deliberately NOT what `SHA256SUMS` covers (only the
  un-prefixed `/install.sh` is checksummed).

**`HONEYCOMB_INSTALL_POSTHOG_KEY`** (optional repository **variable**, not a secret — PostHog
project keys are safe client-side by design): if set, `build.mjs` bakes it into both served scripts
so `install_started` / `install_completed` / `install_failed` (PRD-002c) actually report; if unset,
both scripts' telemetry calls are a documented no-op and the installer still works fully. Configure
at Repo → Settings → Secrets and variables → Actions → **Variables** → New repository variable.

## Why `site/` never ships in the npm tarball

The npm package uses a `files` **allowlist** in `package.json` — only the paths it names are
packed. `site/` is **not** in that allowlist, so the Pages project, this README, the template,
the Function, and `dist/` are all excluded from `npm pack` / publish by construction (verified
with `npm pack --dry-run`). No `.npmignore` entry is needed; the allowlist is the gate.
