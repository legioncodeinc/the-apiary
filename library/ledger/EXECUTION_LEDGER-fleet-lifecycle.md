**Branches:** honeycomb `feature/fleet-lifecycle`; nectar `feature/fleet-lifecycle`; doctor `feature/fleet-lifecycle`; hive `feature/fleet-lifecycle`; superproject `feature/smoker-fleet-lifecycle` (this ledger + scripts/install + site/install).

**Orchestrator decisions (PRD open questions adopted as flagged/proposed, recorded here at run start):**

1. **Signal precedence (003a):** any of the three fleet signals present (registry entry, live 3853 answer, npm global) means FLEET for popup-suppression purposes; the classification logs which signals fired. Suppressing wrongly is cheap; popping wrongly is the bug.
2. **Evaluation moment (003a):** live at each daemon start / login decision, not persisted at install. Self-heals after a hive uninstall.
3. **`nectar login` mechanism (003a):** in-process device-flow client (required for nectar-only solo installs); on-disk result byte-compatible with honeycomb's credentials-store shape.
4. **Verb spelling (003b):** bare `start` / `stop` / `uninstall` everywhere; every existing spelling (`daemon start|stop|status`, `service-status`, `install-service` / `uninstall-service`) keeps working as an alias.
5. **Uninstall vs npm package (003b):** per-product `uninstall` does NOT remove the npm global package (fragile mid-command); package removal belongs to purge (003c) and the script (003d).
6. **hive `stop` semantics (003b):** immediate stop; no active-session warning gate in this iteration.
7. **Purge confirmation strength (003c):** the interactive gate requires typing `purge` (not y/N); `--yes` is the only non-interactive bypass.
8. **Legacy name inventory (003c/003d):** code-derived during recon from each repo's `SERVICE_LABEL` / legacy constants and migration code; frozen in the "Coverage inventory" section below; doctor purge and the scripts consume the same frozen list (mirrored, never imported, per fleet convention).
9. **Windows self-removal (003c):** doctor's final self npm-uninstall runs as a detached best-effort step; the uninstall script (003d) is the documented fallback when it fails.
10. **Script self-containment (003d):** the uninstall scripts are always fully self-contained (no `doctor purge` delegation dual-path); service and dir removal never require Node.
11. **Non-TTY without `--yes` (003d):** hard refuse with instructions (pass `--yes` or download-then-run); never proceed after a delay, never hang. Confirmation reads from `/dev/tty` when available.
12. **Route spelling (003d):** all three routes ship (`/uninstall` serving shell, `/uninstall.sh`, `/uninstall.ps1`), all inside the `SHA256SUMS` guarantee.
13. **sudo/system-scope units (003d):** the script does not escalate; system-scope units it cannot remove are reported with the exact command to run.
14. **ADR-0004 (index open question):** deferred out of this run; noted for a documentation pass. No AC depends on it.
