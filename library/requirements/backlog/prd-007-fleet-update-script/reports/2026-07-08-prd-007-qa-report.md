## Summary

**SHIP.** All 38 acceptance criteria now pass, verified live on this machine. The core package-update/service-restart flow (007a) resolves blessed targets correctly, skips the already-current `honeycomb` package, and previews exactly the moves and telemetry the PRD specifies (confirmed via `--dry-run` on both `update.sh` and `update.ps1` under Windows PowerShell 5.1). The site delivery surface (007c) is fully correct and independently verified (`sha256sum -c` on all six scripts, clean `node build.mjs`, `/update` worker route mirroring `/uninstall`, unchanged deploy workflow). The two defects found in the first audit are both resolved and re-verified:

1. **The Critical harness-verb bug (b-AC-1 / b-AC-6) is fixed.** Both scripts now call the real, verified verb `honeycomb harness status` (preferring `--json`, PRD-006c/006d), via new `detect_harnesses()` (`update.sh:599-668`) and `Get-DetectedHarnesses()` (`update.ps1:496-545`) that JSON-parse the output (empty array/object → "none" clean no-op; non-empty → "detected" list; non-JSON plain text on today's blessed build → treated as the already-human-readable report). No dead `honeycomb harnesses` call remains anywhere — grep confirms the only surviving occurrences are in explanatory comments. Live on this machine: `honeycomb harness status --json` returns `claude-code: agent present, plugin enabled.` (exit 0), which the parser correctly routes to the "detected" branch; the previously-unreachable b-AC-6 "none" branch is now reachable because it no longer depends on an exit code that never happened.
2. **The PS1 exit-code Warning is resolved.** `update.ps1`'s entrypoint now mirrors `uninstall.ps1`'s `IsTopLevelFileInvocation` pattern (`update.ps1:738-751`), so `-File` invocations propagate the process exit code while the documented `irm | iex` bootstrap still hands control back without tearing down the caller's session. Verified live: `powershell -File update.ps1 --bogus` now returns process exit 1 (was 0); `--dry-run` returns 0.

No regression to the 34 previously-passing behaviors: a re-run `--dry-run` produced byte-identical output to the first audit, and `install.sh`/`uninstall.sh` remain untouched and still build/checksum correctly.

## Suggestions (consider improving)

- [ ] **`detect_harnesses` fallback text names `honeycomb harness status`, which may not exist on an even older honeycomb pin**, `scripts/install/update.sh:704`, `scripts/install/update.ps1:581`

  The `unknown`-state fallback branch now correctly guides to `honeycomb harness status` (the real verb on the blessed build). For extra future-proofing — since the update script's whole purpose is bridging version gaps and could resolve a machine sitting on a honeycomb pin that predates the `harness` verb entirely — consider a version-agnostic phrasing ("run `honeycomb --help` to see available commands"). Not required for ship; the `hive-release.json`-pinned version is the only supported target and the branch is already fail-soft (never blocks the update). Low priority.
